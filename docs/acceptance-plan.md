# CloudMall 验收计划

> 版本：v1.0 | 日期：2026-07-04

本文档定义每个开发阶段结束时的验收检查清单，包括页面、接口、数据、正常场景和异常场景的检查项。

---

## 阶段 0：项目骨架搭建

### 0.1 检查节点

| 节点 | 地址 | 检查内容 |
|------|------|----------|
| Swagger UI | `http://localhost:8080/swagger-ui.html` | API 文档页面可访问，展示所有已注册接口 |
| 健康检查 | `http://localhost:8080/api/v1/health` | 返回 `{"code":200,"message":"success","data":"OK"}` |
| Actuator | `http://localhost:8080/actuator/health` | 返回各组件健康状态（DB/Redis/RabbitMQ） |

### 0.2 检查接口

| 方法 | 接口 | 预期状态码 | 预期响应 |
|------|------|------------|----------|
| GET | `/api/v1/health` | 200 | `{"code":200,"data":"OK"}` |
| GET | `/swagger-ui.html` | 200 | Swagger 页面 HTML |
| GET | `/v3/api-docs` | 200 | OpenAPI JSON |

### 0.3 检查数据
- MySQL：`cloudmall` 数据库自动创建，`flyway_schema_history` 表存在
- Redis：`PING` → `PONG`
- RabbitMQ：管理控制台 `http://localhost:15672` 可访问（guest/guest）

### 0.4 正常场景
1. `mvn clean install` 编译通过，所有模块构建成功
2. `mvn test` 测试全部通过（包含 ArchUnit 架构测试）
3. 应用启动日志无 ERROR
4. 多环境配置切换：`dev` / `prod` 启动均正常

### 0.5 异常场景
1. 数据库不可用 → 应用启动失败，日志清晰说明原因
2. Redis 不可用 → 应用启动失败或降级启动
3. 端口被占用 → 启动失败，明确报错

### 1.2 检查接口

| 方法 | 接口 | 预期 | 检查点 |

| POST | `/api/v1/auth/login` | 200 | 返回 AccessToken + RefreshToken |
| POST | `/api/v1/auth/refresh` | 200 | 返回新 AccessToken |
| GET | `/api/v1/users/me` | 200 | 返回当前用户信息（带 JWT） |
| GET | `/api/v1/users/me` | 401 | 无 Token 返回 401 |
| PUT | `/api/v1/users/me` | 200 | 更新昵称/头像成功 |
| GET | `/api/v1/users/me/addresses` | 200 | 返回地址列表 |

| POST | `/api/v1/admin/roles` | 201 | 创建角色 |
| POST | `/api/v1/admin/roles/{id}/permissions` | 200 | 分配权限 |
| POST | `/api/v1/auth/logout` | 200 | Token 加入黑名单 |

### 1.3 检查数据

| 表 | 检查项 |
|------|------|
| `user` | 注册后记录存在，密码字段 BCrypt 加密 |
| `account` | 用户名、加密密码正确 |
| `role` | 预置 ROLE_USER, ROLE_ADMIN |
| `permission` | 预置基础权限 |
| `user_address` | 地址 CRUD 正确 |
| `login_record` | 每次登录新增一条记录 |

### 1.4 正常场景

| 场景 | 步骤 | 预期结果 |
|------|------|----------|
| 注册→登录→访问 | 1. 注册 2. 登录获取 Token 3. 用 Token 访问 `/users/me` | 返回用户信息 |
| 权限隔离 | 1. 普通用户登录 2. 访问 `/admin/users` | 返回 403 Forbidden |
| 管理员管理用户 | 1. 管理员登录 2. 查看用户列表 3. 封禁用户 | 管理员可操作用户状态 |
| Token 刷新 | 1. 登录 2. 用 RefreshToken 刷新 | 获取新 AccessToken |
| 登出 | 1. 登录 2. 登出 3. 再用旧 Token 访问 | 返回 401 |

### 1.5 异常场景

| 场景 | 预期 |
|------|------|
| 重复邮箱注册 | 返回 `{"code":40900,"message":"邮箱已存在"}` |
| 密码不足 8 位 | 返回 `{"code":40001,"message":"密码长度不能少于8位"}` |
| 邮箱格式错误 | 返回 `{"code":40001,"message":"邮箱格式不正确"}` |
| 登录密码错误 | 返回 `{"code":40101,"message":"用户名或密码错误"}` |
| Token 过期 | 返回 `{"code":40102,"message":"令牌已过期"}` |
| Token 被篡改 | 返回 `{"code":40103,"message":"令牌无效"}` |
| 冻结用户登录 | 返回 `{"code":40104,"message":"账户已被冻结"}` |
| 修改密码（旧密码错误） | 返回 `{"code":40001,"message":"旧密码不正确"}` |
| 地址超过 20 个 | 返回 `{"code":40001,"message":"收货地址最多20个"}` |

## 阶段 3：库存中心

### 3.1 检查接口

| 方法 | 接口 | 预期 |
|------|------|------|
| POST | `/api/v1/admin/inventory/initialize` | 200（初始化库存） |
| POST | `/api/v1/admin/inventory/inbound` | 200（入库） |
| GET | `/api/v1/admin/inventory/{skuId}` | 200（查询库存） |
| GET | `/api/v1/admin/inventory/logs/{skuId}` | 200（库存日志） |

### 3.2 正常场景

| 场景 | 预期 |
|------|------|
| 初始化库存 100 → 查询 | 可用库存 = 100 |
| 入库 50 → 查询 | 可用库存 = 150 |
| 扣减 10 → 查询 | 可用库存 = 140, 锁定库存 = 10 |
| 释放 10 → 查询 | 可用库存 = 150, 锁定库存 = 0 |
| 库存变更日志 | 每次操作都有日志记录（操作类型、变更量、操作人） |

### 3.3 异常场景（核心）

| 场景 | 预期 |
|------|------|
| **并发扣减不超卖** | 100 个线程各扣 1，库存 100 → 最终库存 = 0，成功扣减 = 100 次 |
| 库存不足时扣减 | 返回 409，库存不变 |
| 库存预警 | 库存低于安全阈值 → 触发 StockLowEvent |

## 阶段 7：订单中心

### 7.1 检查接口

| 方法 | 接口 | 预期 |
|------|------|------|
| POST | `/api/v1/orders` | 201（创建订单） |
| GET | `/api/v1/orders` | 200（订单列表、分页、状态筛选） |
| GET | `/api/v1/orders/{id}` | 200（订单详情） |
| POST | `/api/v1/orders/{id}/cancel` | 200（取消订单） |
| POST | `/api/v1/orders/{id}/confirm-receive` | 200（确认收货） |
| GET | `/api/v1/admin/orders` | 200（后台订单列表） |

### 7.2 正常场景（核心）

| 场景 | 步骤 | 预期 |
|------|------|------|
| 完整下单流程 | 1. 选择商品 + 地址 + 优惠券 2. 提交订单 | 订单创建，库存扣减，优惠券冻结 |
| 支付→发货→收货 | 模拟支付成功 → 发货 → 确认收货 | 状态依次流转：PAID → DELIVERED → RECEIVED |
| 取消未支付订单 | 待支付订单取消 | 库存释放，优惠券恢复 |
| 订单超时自动取消 | 下单后 30 分钟未支付 | 订单自动取消，库存释放 |
| 订单地址快照 | 下单后修改收货地址 | 订单中的地址不变（快照已保存） |
| 订单商品快照 | 下单后商品涨价 | 订单中价格不变 |

### 7.3 异常场景

| 场景 | 预期 |
|------|------|
| 库存不足时下单 | 返回 409 "库存不足" |
| 商品已下架时下单 | 返回 400 "商品已下架" |
| 重复提交订单 | 幂等防重，返回已创建订单 |
| 取消已支付订单 | 返回 400（需走售后流程） |
| 确认收货未发货订单 | 返回 400 "订单尚未发货" |
| 超过自动取消时间取消 | 返回 400 "订单已自动取消" |

### 7.4 订单状态机验证

```
PENDING_PAY ──支付成功──→ PAID
     │                      │
     │ (超时/主动)           │ (发货)
     ↓                      ↓
 CANCELLED              DELIVERED
                            │
                            │ (确认收货)
                            ↓
                        RECEIVED
                            │
                            │ (自动/手动)
                            ↓
                        COMPLETED
```

**检查点**：每个状态只能从合法的前置状态进入。

## 验收流程规范

每个阶段完成时，按以下顺序验收：

```
1. 编译检查 → mvn clean install
2. 测试检查 → mvn test (含单元测试 + 集成测试 + 架构测试)
3. 启动检查 → 应用无 Error 启动
4. Swagger 检查 → 接口文档与实现一致
5. 接口检查 → Postman / curl 逐个验证
6. 数据检查 → 数据库中验证数据正确性
7. 页面检查 → 前端页面功能走查
8. 正常场景 → 端到端走一遍完整流程
9. 异常场景 → 验证错误处理正确
10. 回归检查 → 之前阶段的功能未被破坏
```
